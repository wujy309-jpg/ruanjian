# 知识库设计说明书

> 本文档给 B 使用，但 A 也需要理解知识图谱的结构才能写 PlanAgent 和 GenAgent 的 prompt。

---

## 一、核心设计思路

知识库不是传统题库，而是一个 **课程知识图谱**。它的作用是让 Agent 知道：

- 这门课有哪些知识点
- 知识点之间的依赖关系（先学 A 才能学 B）
- 每个知识点的难度
- 每个知识点的详细描述（大模型靠这个理解"这个知识点是什么"）

**一句话**：知识图谱 = PlanAgent 的地图 + GenAgent 的说明书。

## 二、数据结构

### 2.0 课程表 `course`

```sql
CREATE TABLE course (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(200) NOT NULL COMMENT '课程名称',
    description TEXT COMMENT '课程描述',
    status      ENUM('draft','published') DEFAULT 'draft' COMMENT '状态',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
) COMMENT '课程表';
```

### 2.1 知识点表 `knowledge_point`

```sql
CREATE TABLE knowledge_point (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    course_id   BIGINT NOT NULL COMMENT '所属课程ID',
    parent_id   BIGINT DEFAULT NULL COMMENT '父知识点ID（树形分级，可空）',
    name        VARCHAR(200) NOT NULL COMMENT '知识点名称（给人看的）',
    code        VARCHAR(50) NOT NULL COMMENT '知识点编码（给程序用的，唯一）',
    difficulty  ENUM('L1','L2','L3') NOT NULL DEFAULT 'L1' COMMENT '难度层级',
    description TEXT COMMENT '知识点详细描述（给大模型看的，这部分最重要）',
    keywords    VARCHAR(500) COMMENT '关键词，逗号分隔，用于检索',
    sort_order  INT DEFAULT 0 COMMENT '排序',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,

    KEY idx_course (course_id),
    KEY idx_parent (parent_id),
    UNIQUE KEY uk_code (code)
) COMMENT '知识点表';
```

### 2.2 前置依赖表 `knowledge_edge`

```sql
CREATE TABLE knowledge_edge (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    from_kp_id  BIGINT NOT NULL COMMENT '前置知识点（必须先掌握）',
    to_kp_id    BIGINT NOT NULL COMMENT '目标知识点（依赖前置）',
    edge_type   VARCHAR(20) DEFAULT 'requires' COMMENT '关系类型',
    -- requires: 必须先学会from才能学to
    -- suggests: 建议先学from再学to（不强制）

    UNIQUE KEY uk_edge (from_kp_id, to_kp_id)
) COMMENT '知识点前置依赖关系';
```

**关系语义**：

```
[类与对象] ──requires──→ [构造方法]
     │                       ↑
     └───────requires──→ [继承与多态]
                             ↑
           [接口] ──requires──┘

含义：学"构造方法"前必须先掌握"类与对象"
     学"继承与多态"前必须先掌握"类与对象"
     学"接口"是"继承与多态"的延伸
```

## 三、知识图谱的层级结构

```
课程 (Course)                       ← 一门课一个知识图谱
 └── 章节 (Chapter)                 ← 逻辑分组（通过 parent_id 实现）
      ├── 知识点 (KnowledgePoint)   ← 最小学习单元
      │    ├── difficulty: L1/L2/L3
      │    ├── description: 给 Agent 看的
      │    └── keywords: 检索用
      └── 知识点之间的依赖边 (Edge)  ← DAG（有向无环图）
```

**为什么用 parent_id 做树形而不是单独建 chapter 表**：
- 知识点的粒度已经够细了，"章节"只是一个分组标签
- 如果某知识点跨章节（比如"面向对象思想"既在第一章又在第二章被引用），parent_id 可以留空，让 edge 表来表达跨章节依赖
- 减少表数量，降低复杂度

## 四、知识点的完整数据结构（JSON 视图）

这是 B 在构建知识库时每一个知识点的完整信息。A 的 PlanAgent 和 GenAgent 会以这个格式消费：

```json
{
  "id": 1,
  "code": "java-oop-01",
  "name": "类与对象",
  "parent_code": "java-ch01",
  "difficulty": "L1",
  "keywords": ["类", "对象", "实例化", "成员变量", "成员方法", "new关键字"],
  "description": "## 核心概念\n类是Java中对象的模板（蓝图），定义了对象的属性（成员变量）和行为（方法）。对象是类的具体实例，通过new关键字创建。理解类与对象的关系是面向对象编程的第一步。\n\n## 知识点范围\n- 类的定义语法（class关键字）\n- 成员变量（属性）的声明\n- 成员方法（行为）的定义\n- 使用new关键字创建对象\n- 通过对象引用访问属性和方法\n- 栈内存与堆内存中对象的存储模型\n\n## 前置知识\n无（这是Java面向对象的第一课）\n\n## 常见误区\n- 混淆类和对象：类是抽象的模板，对象是具体的实例\n- 以为一个类只能创建一个对象\n- 不理解对象引用变量和对象的区别\n- 误以为方法必须写在main方法里",
  "prerequisites": [],
  "children": [
    {
      "code": "java-oop-02",
      "name": "构造方法"
    },
    {
      "code": "java-oop-04",
      "name": "this关键字"
    }
  ]
}
```

**关键字段说明**：

| 字段 | 谁用 | 怎么用 |
|------|------|--------|
| `code` | A & B | 唯一标识，edge 表和 API 传参都靠它 |
| `name` | C | 前端展示 |
| `difficulty` | A (PlanAgent) | 路径规划时判断难度递进 |
| `keywords` | A (GenAgent) | 检索已有资源，避免重复生成 |
| `description` | A (所有Agent) | **最重要的字段**，Agent 靠它理解知识点内容 |
| `prerequisites` | A (PlanAgent) | 路径排序的硬约束 |

## 五、description 字段的撰写规范

这是知识库最难、也最重要的部分。写得好不好直接决定 Agent 的生成质量。

**规则**：

1. **至少 150 字**，用 Markdown 格式，分节清晰
2. **必须覆盖**：核心概念、知识点范围、前置知识、常见误区
3. **不要写**：具体代码示例（让 GenAgent 自己生成，写死了反而不个性化）
4. **要写**：教学边界（什么属于这个知识点，什么不属于）

**好的 description vs 差的 description**：

```
❌ 差：构造方法是类中用来创建对象的方法。

✅ 好：
## 核心概念
构造方法是创建对象时自动调用的特殊方法。方法名必须与类名完全相同，没有返回值类型（连void都没有）。每个类至少有一个构造方法：如果没写，编译器自动添加无参构造方法；如果写了任何构造方法，默认构造方法消失。

## 知识点范围
- 构造方法的定义语法
- 默认构造方法与自定义构造方法的关系
- 构造方法的重载（多个构造方法，参数列表不同）
- 在构造方法中使用this()调用另一个构造方法
- 构造方法中的初始化代码

## 前置知识
必须先理解"类与对象"的基本概念（java-oop-01），否则无法理解构造方法在对象创建流程中的作用。

## 常见误区
- 以为构造方法有返回值（它连void都不是）
- 不理解为什么写了有参构造方法后new 类名()会报错
- this()调用另一个构造方法时错误放在非首行
- 混淆构造方法和普通成员方法的调用时机
```

## 七、知识库的查询 API

### 7.1 完整图谱查询

```
GET /api/knowledge/graph/{courseId}

用途：PlanAgent 拿到全部知识点和依赖关系，做全局路径规划
```

### 7.2 单个知识点查询

```
GET /api/knowledge/points/{id}

用途：GenAgent 拿到某个知识点的 description，生成该知识点的学习资源
```

### 7.3 按知识点编码查询

```
GET /api/knowledge/points/code/java-oop-01

用途：GenAgent 生成题目时，通过 code 确定题目属于哪个知识点
```

### 7.4 获取某知识点的前置依赖链

```
GET /api/knowledge/points/{id}/prerequisite-chain

用途：PlanAgent 验证用户画像中的知识掌握度是否覆盖了目标知识点的所有前置
```

### 7.5 树形结构查询（前端用）

```
GET /api/knowledge/tree/{courseId}

用途：C 在学习页面展示知识点的层级导航
返回格式：
{
  "courseId": 1,
  "nodes": [
    {
      "code": "java-basic-01",
      "name": "Java语言概述",
      "children": [
        { "code": "java-basic-02", "name": "HelloWorld", "children": [] }
      ]
    }
  ]
}
```

## 八、PlanAgent 如何使用知识图谱

PlanAgent 的核心逻辑是一个拓扑排序的变体：

```
输入：
  1. 知识图谱（所有知识点 + 依赖边）
  2. 用户画像（knowledge_map 中各知识点的掌握度）

算法：
  1. 把用户已掌握（level >= 0.6）的知识点标记为 done
  2. 把用户薄弱（level < 0.3）的知识点标记为 focus
  3. 从所有 done 节点出发，找"所有前置都满足、且自身未完成"的节点 → 这些是下一步候选
  4. 候选节点中，focus 类型优先排，review 类型穿插排
  5. 按 difficulty 递进排列
  6. 为每个节点预估学习时长（L1=20min, L2=35min, L3=50min，再根据用户风格调整）
  7. 输出不超过 25 个节点的路径

输出：LearningPath（路径节点序列，含每个节点的类型、时长、理由）
```

## 九、GenAgent 如何使用知识图谱

GenAgent 拿到一个知识点编码后：

```
1. 查 GET /api/knowledge/points/code/{code}
2. 拿到该知识点的 name、description、difficulty、keywords
3. 根据 description 中的"知识点范围"确定生成内容的边界
4. 根据 difficulty 确定生成内容的深度（L1=入门、L2=进阶、L3=深度）
5. 根据 keywords 检索已有资源（避免重复生成）
6. 调用大模型生成：
   - document：用 description 的 Markdown 大纲扩展为完整讲解
   - quiz：根据"常见误区"出判断题，根据"知识点范围"出选择题
   - mindmap：把 description 的层级结构转为树形 JSON
```

## 十、B 的工作步骤总结

1. **第1步**：建表（B1，1天）— 跑 DDL
2. **第2步**：按 6.2 的知识点清单，为每个知识点写 description（B2，2天）
   - 先在 YAML 或 Excel 里写，不要直接写 SQL
   - 每个 description ≥ 150 字，按第五章的规范
3. **第3步**：写 SQL 导入脚本或 YAML 解析器（B2 的一部分）
4. **第4步**：写查询 API（B7，2天）
5. **第5步**：在 API 返回结果里联表查 edge 表，拼出完整的 prerequisites 列表

**最耗时的是第 2 步**——40 个知识点 × 150 字 = 6000 字的内容撰写。建议 B 优先启动。
